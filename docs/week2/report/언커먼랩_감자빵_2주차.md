# 감자빵 2주차 보고서 - 문서 카테고리 추천을 위한 LLM 탐색 및 프롬프트 구현

본 프로젝트의 파이프라인(OCR → sLLM 카테고리 분류)에서  
OCR 결과 텍스트를 입력받아 문서 카테고리를 추천하는 Instruct LLM 후보를 선정하고,  
카테고리 분류를 위한 한글/영문 프롬프트를 구현한다.

조사 기준일: 2026년 4월 23일

---

## 1. 문서 카테고리 추천을 위한 LLM 탐색

### 1.1. 선정 기준

| 기준 | 요구사항 |
|---|---|
| 모델 유형 | **Instruct(Non-thinking) 모델** — Thinking 없이 즉시 응답 |
| 응답 속도 | 서비스 기준 **5초 이내** (페이지당 전체 파이프라인 포함) |
| 태스크 | 문서 카테고리 분류 (용선계약서, 사고보고서, 정산서, 근로계약서 등 단일 라벨 출력) |
| 대상 도메인 | 물류 — 상사계약, 운영·관리, 보험·사고, 재무, 일반 행정·HR 등 |
| 다국어 | 한국어·영어 혼용 문서 처리 가능 (물류 문서는 한영 혼용이 빈번) |
| 실행 환경 | RTX 3060 (12GB VRAM), Ollama 로컬 실행 |
| 출력 형식 | 카테고리명만 단답 출력 (불필요한 설명 최소화) |
| 우선순위 | **속도 > 정확도 > 출력 형식 준수** |

#### Instruct 모델을 사용해야 하는 이유

1. **속도**: Thinking 모델은 내부 추론 과정이 추가되어 토큰 생성량이 증가하고 응답 지연이 커짐. 카테고리 분류는 단순 분류 태스크로 복잡한 chain-of-thought가 불필요함.
2. **출력 제어**: Instruct 모델은 "카테고리명만 출력하라"는 지시를 잘 따름. Thinking 모델은 reasoning 과정을 억제해도 불필요한 설명이 포함되는 경우가 많음.
3. **비용 효율**: 동일 VRAM에서 Instruct 모델이 더 높은 throughput을 달성하여 동시 요청 처리에 유리.

### 1.2. 최종 후보 모델 (3개)

OCR을 통해 추출된 물류 도메인 문서 텍스트를 입력받아 카테고리를 분류하는 모델이다.  
물류 문서는 계약서·정산서 등 한국어 본문에 영문 해운 용어(NYPE, B/L, CII 등)가 혼용되는 특성이 있으므로 한영 이중언어 처리 능력이 중요하다.

#### 1) Qwen3-8B Non-thinking (Alibaba)

| 모델 | 파라미터 | 출시 | 컨텍스트 | 라이선스 |
|---|---|---|---|---|
| Qwen3-8B | 8.2B (비임베딩 6.95B) | 2025년 5월 | 32K (YaRN 확장 128K) | Apache 2.0 |

- Thinking/Non-thinking 모드를 단일 모델 내에서 전환 가능 (`enable_thinking=False`)
- Non-thinking 모드에서 Qwen2.5-Instruct 수준의 빠른 응답 제공
- **100개 이상 언어** 지원, 한/중/일 아시아 언어 처리에서 동급 최강
- 공식 발표 기준 Qwen3-8B는 Qwen2.5-14B급 성능을 달성
- Ollama에서 `ollama run qwen3:8b`로 즉시 실행 가능 (INT4 기본, ~5GB VRAM)

> **선정 근거**: Qwen3 Dense 모델은 0.6B~32B의 6종으로 제공된다.
> 4B는 1주차에서 Qwen3.5-4B로 이미 실험 예정이므로 중복, 14B 이상은 RTX 3060 VRAM 초과.
> 8B는 INT4 양자화 시 ~5GB VRAM으로 RTX 3060에서 여유 있게 구동된다.
>
> 같은 Qwen 계열인 **Qwen2.5-7B-Instruct**도 검토했으나,
> Qwen3-8B가 동일 계열 상위호환(학습 데이터 36T vs 18T, 119개 언어 vs 29개, Qwen2.5-14B급 성능)이므로
> 두 모델을 동시에 넣는 것은 슬롯 낭비로 판단하여 Qwen3-8B만 채택했다.

#### 2) EXAONE 3.5 7.8B Instruct (LG AI Research)

| 모델 | 파라미터 | 출시 | 컨텍스트 | 라이선스 |
|---|---|---|---|---|
| EXAONE-3.5-7.8B-Instruct | 7.8B (비임베딩 6.98B) | 2024년 12월 | 32K | EXAONE AI Model License 1.1 - NC |

- LG AI Research에서 개발한 **한국어·영어 이중언어 특화** 모델
- 한국어 벤치마크: **KoMT-Bench 7.96**, **LogicKor 9.08**로 동급 최고 수준
- 범용 벤치마크: Arena Hard 68.7, IFEval 78.9, MT-Bench 8.29
- 물류 관련 한국어 문서(용선계약서, 정산서, 사고보고서 등)의 문맥 파악에 한영 이중언어 학습이 유리할 것으로 기대
- Ollama에서 `ollama run exaone3.5`로 즉시 실행 가능 (Q4 기본, 4.8GB)
- **NC 라이선스**로 연구/비상용 목적 무료, 상용 적용 시 별도 협의 필요

> **선정 근거**: EXAONE 시리즈는 3.5(2.4B/7.8B/32B)와 4.0(1.2B/32B)이 있다.
> EXAONE 4.0은 7-8B급 모델이 없어 선택 불가, 3.5 2.4B는 파라미터 부족, 32B는 RTX 3060 구동 불가.
>
> 과제에서 제안된 **HyperCLOVA X SEED 8B**도 검토했으나,
> HuggingFace 확인 결과 텍스트 전용 Instruct 8B 모델이 존재하지 않았다
> (Omni-8B은 멀티모달, Text-Instruct는 0.5B/1.5B만 존재).
> 한국어 특화 8B급 텍스트 전용 모델로는 EXAONE 3.5 7.8B가 유일한 선택지이다.

#### 3) Ministral-3-8B-Instruct-2512-BF16 (Mistral AI)

| 모델 | 파라미터 | 출시 | 컨텍스트 | 라이선스 |
|---|---|---|---|---|
| Ministral-3-8B-Instruct-2512-BF16 | 8.4B(LM) + 0.4B(Vision) | 2025년 12월 | 256K | Apache 2.0 |

- 과제에서 도입 검토 모델로 명시된 후보(Ministral3:8b)의 최신판
- LM + Vision Encoder 멀티모달 구조, 본 과제에서는 텍스트 분기만 사용
- Edge 최적화 설계로 BF16 기준 24GB VRAM, 4bit 양자화 시 12GB 미만에 적재 가능(Mistral 공식 문서)
- **Apache 2.0 / non-gated** 라이선스로 상용 적용 시에도 자유롭게 이용 가능

> **선정 근거**: 과제에서 도입 검토 모델로 명시(Ministral3:8b).
> 당초 후보였던 **Ministral-8B-Instruct-2410**은 Mistral Research License + gated repo로 라이선스 동의 절차가 필요하고 상업 적용에 제약이 있다.
> 후속 모델인 2512-BF16은 **Apache 2.0, non-gated**로 접근성·상용 확장성이 우수하며,
> 비전 인코더가 포함된 멀티모달 구조지만 텍스트 전용 추론 경로(`AutoModelForImageTextToText` + 텍스트 입력)로 동일 분류 태스크에 적용 가능하다.
> 라이선스 접근성과 향후 MVP 상용 확장을 우선하여 2512-BF16으로 채택했다.

## 2. 문서 카테고리 추천을 위한 프롬프트 구현

1차 검증은 **은행 도메인 샘플**(AI Hub 공공문서 OCR 데이터 중 은행 분류 40장)로 수행했다.  
**출력 형식은 JSON 단일 라벨**로 간소화하여 파서 자동화 및 모델 간 공정 비교를 우선했다. 한글/영문 2개 버전을 구현했다.

### 2.1. 카테고리 목록 (은행 도메인, 1차 검증용)

| 카테고리 | 정의 | 대표 예시 |
|---|---|---|
| 신고서 | 사건/사실/변경 내용을 은행에 알리기 위해 제출하는 문서 | 분실신고서, 변경신고서 |
| 신청서 | 특정 서비스/상품 가입이나 처리를 요청하는 문서 | 계좌개설신청서, 대출신청서 |
| 위임장 | 본인의 권한을 제3자에게 위임하는 문서 | — |
| 확인서 | 특정 사실/상태를 증명하거나 확인하는 문서 | 잔액확인서, 거래확인서 |


### 2.2. 한글 프롬프트

**System 메시지**

```
당신은 은행 문서를 4개 카테고리 중 하나로 분류하는 전문가입니다.

카테고리 정의:
- 신고서: 사건/사실/변경 내용을 은행에 알리기 위해 제출하는 문서 (예: 분실신고서, 변경신고서)
- 신청서: 특정 서비스/상품 가입이나 처리를 요청하는 문서 (예: 계좌개설신청서, 대출신청서)
- 위임장: 본인의 권한을 제3자에게 위임하는 문서
- 확인서: 특정 사실/상태를 증명하거나 확인하는 문서 (예: 잔액확인서, 거래확인서)

반드시 아래 JSON 형식으로만 응답하십시오. 다른 설명 금지.
{"label": "<카테고리명>"}
```

**User 메시지 템플릿**

```
다음은 OCR로 추출된 은행 문서의 마크다운 내용입니다. 이 문서를 위 4개 카테고리 중 하나로 분류하세요.

--- OCR 내용 시작 ---
{ocr_text}
--- OCR 내용 끝 ---

JSON 형식으로만 응답: {"label": "..."}
```

**출력 예시:**

| 입력 (OCR 마크다운 요약) | 출력 |
|---|---|
| "분실신고서 / 본인은 통장 분실 사실을 신고합니다" | `{"label": "신고서"}` |
| "신용카드 발급 신청서 / 신청인 ○○○" | `{"label": "신청서"}` |
| "위임장 / 위임인 ○○○은 수임인 ○○○에게 계좌 해지 권한을 위임합니다" | `{"label": "위임장"}` |
| "잔액증명서 / 상기 금액은 2026-04-20 기준 잔액임을 확인합니다" | `{"label": "확인서"}` |

### 2.3. 영문 프롬프트

라벨 출력은 한글로 고정하여 파서 로직 및 정답 비교 대상(폴더명)과 일치시킨다. 프롬프트 본문만 영어화.

**System 메시지**

```
You are an expert that classifies Korean bank documents into one of 4 categories.

Category definitions:
- 신고서 (report/notice): A document submitted to notify the bank of an event, fact, or change (e.g. loss report, change-of-information notice).
- 신청서 (application): A document requesting enrollment in or processing of a specific service/product (e.g. account opening form, loan application).
- 위임장 (power of attorney): A document delegating the principal's authority to a third party.
- 확인서 (certificate/confirmation): A document proving or confirming a specific fact or status (e.g. balance certificate, transaction certificate).

Rules:
- Respond with the JSON object below and NOTHING ELSE. No explanation, no markdown fences.
- The value of "label" MUST be one of these exact Korean strings: 신고서, 신청서, 위임장, 확인서.

{"label": "<category>"}
```

**User 메시지 템플릿**

```
Below is the markdown content of a Korean bank document extracted by OCR. Classify it into one of the 4 categories defined above.

--- OCR content start ---
{ocr_text}
--- OCR content end ---

Respond in JSON only: {"label": "..."}
```

### 2.4. 프롬프트 설계 포인트

| 설계 요소 | 설명 |
|---|---|
| System/User 분리 | 카테고리 정의·출력 형식은 System에, 실제 OCR 본문은 User 메시지에 배치. 본문이 길어져도 카테고리 정의가 컨텍스트에 안정적으로 유지됨 |
| JSON 단일 라벨 출력 | `{"label": "..."}` 강제 → Ollama `format=json` 옵션과 결합 시 파서 실패 0건 달성. 설명/이유/신뢰도 필드는 1차 검증에서 제외하고 물류 차수에서 재도입 예정 |
| 라벨 문자열은 한글 고정 | 영문 프롬프트에서도 라벨은 `신고서/신청서/위임장/확인서` 한글 유지. 정답 비교(폴더명)와 파서 로직을 한 번에 통일 |
| OCR 노이즈 명시 | User 메시지에 "OCR로 추출된 마크다운"을 명시 → 오타·공백 이상을 키워드 기반으로 우회하도록 유도 |
| 본문 구분자 | `--- OCR 내용 시작/끝 ---` 마커로 본문 경계를 표시 → 모델이 본문 안 지시문을 프롬프트 지시로 오해(prompt injection 유사)하는 것을 완화 |
| 영문 프롬프트의 한영 병기 | 각 카테고리를 `신고서 (report/notice)` 형태로 병기 → 한국어 라벨 출력 제약과 영문 정의가 모순되지 않게 연결 |
| temperature = 0.0 | 재현성 확보 |

## 3. 실험 구조

```
OCR 추출 텍스트 → Instruct LLM → 카테고리 분류 출력
대상: Qwen3-8B, EXAONE-3.5-7.8B, Ministral-3-8B (2512-BF16)
프롬프트: 한글 / 영문 (2종)
```

### 실행 환경

| 환경 | VRAM | 정밀도 | 실행 가능 모델 |
|---|---|---|---|
| RTX 3060 (로컬) | 12GB | Ollama Q4_K_M / bitsandbytes NF4 | Qwen3·EXAONE 구동 확인, Ministral은 로딩 피크 VRAM 초과 (5.5 참조) |

> Qwen3·EXAONE은 Ollama Q4_K_M 양자화로 RTX 3060 12GB에서 구동. Ministral-3-8B(2512)는 멀티모달 + BF16 풀사이즈(17.8GB) → 4bit 양자화 로딩 중 피크 메모리 이슈로 현 하드웨어에서 미확보  
> 속도 비교는 동일 양자화 조건에서 수행하여 공정성 확보

## 4. 평가 지표

| 지표 | 설명 |
|---|---|
| 카테고리 분류 정확도 | 정답 카테고리 대비 모델 출력 일치율 (Exact Match) |
| 응답 속도 | 입력→출력 완료까지 소요 시간 (ms), **5초 이내 여부** |
| 출력 형식 준수율 | 지정 형식대로 출력하는지 vs 불필요한 설명이 포함되는 비율 |
| 한국어 문서 분류 정확도 | 물류 도메인 한국어 문서에 대한 카테고리 분류 정확도 |
| 한영 혼용 문서 처리 | 한국어 본문 + 영문 해운 용어 혼용 문서의 분류 정확도 |
| 한글/영문 프롬프트 성능 차이 | 동일 문서에 대해 한글 프롬프트 vs 영문 프롬프트의 정확도·속도 차이 |

## 5. 테스트 결과

### 5.1. 테스트 조건

1차 검증용으로 **은행 도메인 샘플**(신고서/신청서/위임장/확인서 각 10장, 총 40장, 한국어)을 사용하여 3 모델 × 2 프롬프트 언어 = 6 조합을 평가했다.  
프롬프트는 보고서의 은행의 카테고리에 맞춘 버전을 사용했다. 

- **데이터**: `data/OCR_sample/은행/` 40장 (PaddleOCR-VL 추출 마크다운)
- **하드웨어**: RTX 3060 12GB (Ollama Q4_K_M / bitsandbytes NF4)
- **생성 설정**: `temperature=0.0`, `max_new_tokens=64`, JSON 강제 출력, Qwen3는 `think=False`
- **측정 방식**: 모델당 warmup 1회 후 본 측정 (첫 샘플 latency 오염 제거)

### 5.2. 모델별 분류 정확도

| 모델 | 한글 프롬프트 | 영문 프롬프트 |
|---|---|---|
| Qwen3-8B (Non-thinking) | **100.0% (40/40)** | 75.0% (30/40) |
| EXAONE 3.5 7.8B | 75.0% (30/40) | 82.5% (33/40) |
| Ministral-3-8B (2512-BF16) | 테스트 실패 (VRAM) | 테스트 실패 (VRAM) |

클래스별 F1 (성공 조합):

| 모델 · 언어 | 신고서 | 신청서 | 위임장 | 확인서 |
|---|---|---|---|---|
| Qwen3-8B · ko | 1.000 | 1.000 | 1.000 | 1.000 |
| Qwen3-8B · en | 0.000 | 0.667 | 1.000 | 1.000 |
| EXAONE 3.5 · ko | 0.000 | 0.667 | 1.000 | 1.000 |

### 5.3. 모델별 응답 속도

| 모델 | 프롬프트 | p50 (ms) | p95 (ms) | 평균 (ms) | 5초 이내 |
|---|---|---|---|---|---|
| Qwen3-8B (Non-thinking) | ko | **1,105** | **2,216** | 1,107 | ✅ |
| Qwen3-8B (Non-thinking) | en | 1,094 | 2,235 | 1,221 | ✅ |
| EXAONE 3.5 7.8B | ko | 3,542 | 7,249 | 4,026 | △ (p95 초과) |
| EXAONE 3.5 7.8B | en | 4,916 | 10,567 | 5,321 | ❌ |
| Ministral-3-8B (2512-BF16) | — | — | — | — | 측정 불가 |

- Qwen3은 두 언어 모두 **p95에서도 5초 이내**로 실시간성 확보
- EXAONE은 한국어에서 p50 기준 5초 이내이나 p95에서 초과. 영어 프롬프트에서는 p50도 5초에 근접
- Qwen3 vs EXAONE 동일 Ollama Q4_K_M 환경에서 **Qwen3이 p50 기준 3.2~4.5배 빠름**

### 5.4. 출력 형식 준수율

본 테스트는 JSON 단일 라벨(`{"label": "..."}`) 형식으로 간소화 실행. 형식 준수 = 파서가 유효 라벨을 추출할 수 있는 비율로 측정.

| 모델 | 한글 프롬프트 | 영문 프롬프트 |
|---|---|---|
| Qwen3-8B (Non-thinking) | 100% (40/40) | 100% (40/40) |
| EXAONE 3.5 7.8B | 100% (40/40) | 100% (40/40) |
| Ministral-3-8B (2512-BF16) | — | — |

- Ollama `format=json` 옵션으로 JSON 출력을 강제한 결과, 성공한 2모델 모두 **parse 실패 0건**
- 보고서 2장에서 정의한 구조화 텍스트 형식(`카테고리: ... / 이유: ... / 신뢰도: ...`)에 대한 준수율은 본 차수에서 미측정 — 물류 데이터셋 + 원본 프롬프트 적용 차수에서 검증 예정

### 5.5. 한글/영문 프롬프트 성능 비교 분석

**핵심 관찰**

1. **프롬프트 언어 효과는 모델 의존적**
   - Qwen3-8B: 한글(100%) > 영문(75%) — 한글 프롬프트가 압도적 우위
   - EXAONE 3.5: 영문(82.5%) > 한글(75%) — 한국어 특화 모델임에도 역전 현상
   - → "한국어 특화 모델 = 한글 프롬프트 유리"라는 직관이 항상 성립하지 않음을 시사. 샘플 40장 규모이므로 확장 데이터셋에서 재검증 필요

2. **"신고서 → 신청서" 오분류가 지배적 에러**
   - 4개 성공 조합 중 3개가 신고서 10장의 대부분 또는 전부를 신청서로 분류
   - Qwen3 한글 조합에서만 이 혼동이 완전히 해소됨
   - 위임장 / 확인서는 전 조합에서 F1 = 1.000 (시각·의미 구분 명확)
   - → OCR 결과 내 레이아웃/표제어가 신고서↔신청서 간에 유사하여, 프롬프트의 카테고리 정의 해석 정확도가 성능을 좌우

3. **속도 측면 프롬프트 언어 영향은 제한적**
   - Qwen3: ko/en p50 차이 11ms (1% 수준)
   - EXAONE: 영문이 한글 대비 p50 1,374ms 추가 지연 (39% 증가). 영문 출력 토큰 수 증가 + 한국어 라벨 생성 시 혼합 토크나이징 비용으로 추정

**결론 및 후속 과제**

- 현 시점 최우수 조합: **Qwen3-8B + 한글 프롬프트** (정확도 100%, p50 1.1초, p95 2.2초)
- EXAONE은 속도 측면에서 서비스 5초 요건을 안정적으로 충족하지 못해 현 하드웨어에서는 부적합
- Ministral은 RTX 3060 12GB에서 로컬 실행 미확보 (상세 5.5)
- 영문 OCR 입력 데이터를 확보하여 모델(3) * 프롬프트(2) * 데이터(2) = 12 가지의 조합을 테스트

### 5.5. Ministral-3-8B-Instruct-2512-BF16 테스트 실패

채택한 Ministral-3-8B-Instruct-2512-BF16은 현 실험 환경(RTX 3060 12GB)에서 로컬 실행을 확보하지 못했다.

**시도 경과**

1. HuggingFace에서 BF16 풀사이즈 17.8GB 가중치 다운로드 성공
2. 해당 모델은 LM 8.4B + Vision Encoder 0.4B 멀티모달 구조(Mistral3Config)로, `AutoModelForImageTextToText`로 로딩 필요
3. transformers + bitsandbytes NF4 4bit 양자화 적용, `device_map={"": 0}`로 GPU 단일 배치
4. 모델 weight 로딩 중 약 30% 지점(161/531)에서 **hang 발생**. Ollama 모델 사전 언로드, vision 양자화 포함/제외 등 조치 후에도 재현

**원인 추정**

- BF16 풀사이즈 17.8GB 파일을 레이어 단위로 읽어 4bit 양자화하는 과정에서, 특정 레이어의 bf16 원본과 양자화 사본이 일시적으로 공존하며 **12GB VRAM 피크 초과**
- 멀티모달 구조로 인한 초기 메모리 오버헤드 추가

**대응안**

- 24GB+ GPU(A100 / RTX 4090 등) 확보 시 재시도
- Mistral 공식 사전양자화 체크포인트(FP8 등) 또는 커뮤니티 GGUF 배포분 확보 시 Ollama 등록 후 재측정
- 상업 적용 제약을 감수한다면 대안 후보인 Ministral-8B-Instruct-2410(텍스트 전용, Mistral Research License) 라이선스 동의 후 재시도도 고려 가능
