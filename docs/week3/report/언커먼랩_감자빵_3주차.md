2주차에서 은행 도메인 샘플로 모델 선정 및 한글/영문 프롬프트 1차 검증을 완료했다.  
3주차에서는 멘토 피드백을 반영하여 프롬프트 로직을 고도화하고, 물류 도메인 데이터를 수집하여 Edge Case 테스트 기반을 구축했다.

## 2. 프롬프트 고도화

### 2.1. 설계 원칙

2주차 프롬프트(JSON 라벨 출력)에서 멘토 피드백을 반영하여 다음 원칙으로 재설계했다.

| 원칙 | 설명 |
|---|---|
| 카테고리명만 출력 | Reason, Confidence, Secondary 등 부가 정보 제거. 토큰 절약 + 파싱 간소화 |
| Decision Tree 강제 | `Ambiguity Check → Existing Match → New Category` 순서를 명시하여 모델이 단계를 건너뛰지 않도록 함 |
| 새 카테고리 생성 규칙 | 2-6 단어, 도메인 적합, 기존 카테고리와 배타적, "기타/미분류" 같은 회피성 명명 금지 |
| OCR 노이즈 대응 | 개별 깨진 문자가 아닌 전체 맥락·핵심 키워드 기반 판단 유도 |
| 구조 통일 | 모든 프롬프트를 `[SECTION]` 태그 방식으로 통일하여 변인 통제 |

### 2.2. 프롬프트 3종 비교

물류 도메인 4카테고리(상업송장, 포장명세서, 선하증권, 원산지증명서)에 대한 프롬프트를 3종 설계했다. 5번째 LABEL로 `기타`를 두는 안도 1차 실험에서 검증했지만, 모델이 catch-all 금지 규칙과 충돌해 `기타`를 회피하는 부작용이 컸다(자세한 내용은 4·5장 참조). 따라서 최종 메인 실험용 프롬프트는 4카테고리 + 새 카테고리 제안 + 분류 불가의 3분기 구조로 확정했다.

| 프롬프트 | 출력 언어 | 출력 형식 | 용도 |
|---|---|---|---|
| **Category-Only (영문)** | English | 카테고리명 한 줄 | **메인 실험용** |
| **한국어 출력** | 한국어 | 카테고리명 한 줄 | 영문 대비 한국어 출력 비교용 |
| **상세 버전** | English | Category + Reason + Confidence | 디버깅·분석 전용 (실험 제외) |

#### Category-Only 프롬프트 (메인)

```
[ROLE]
You are an expert document classifier specializing in international logistics and trade documents.

[TASK]
Classify the attached document according to the [CLASSIFICATION RULES] and [CATEGORY LIST] below, then output the result strictly in the format defined in [OUTPUT FORMAT].

[PROCESSING ORDER]
Follow this decision tree in strict order. Stop at the first step that yields a valid result.
Step 1 — Ambiguity Check: If the document lacks identifiable subject cues (no document-type noun, proper noun, action verb, or domain keyword), output "Unclassifiable".
Step 2 — Existing Category Match: Map the document's core purpose to one of the four existing categories. If any reasonable match exists, output that category name.
Step 3 — New Category Proposal: Only if Step 2 yields no plausible match, output a newly proposed category name.

[OCR HANDLING]
The source may contain OCR errors. Base the decision on dominant keywords and overall semantic context, not on isolated garbled tokens. Ignore single corrupted characters when the surrounding context is clear.

[CLASSIFICATION RULES]
1. Identify the document's core purpose and select the single most relevant category.
2. Output exactly ONE category. Do not list secondary categories, even if the document touches multiple areas. Choose the dominant purpose.
3. Propose a new category ONLY when none of the four existing categories covers the core purpose. When uncertain, choose the closest existing category instead of inventing one.
4. New-category naming rules (must satisfy ALL):
   a. Noun phrase, 2–6 words.
   b. Domain-appropriate (international logistics, trade, customs, or shipping context).
   c. Mutually exclusive with all existing categories.
   d. General enough to cover similar future documents — never document-specific or vendor-specific.
   e. Avoid catch-all names such as "Other", "Miscellaneous", "General".

[CATEGORY LIST]
- 상업송장
- 포장명세서
- 선하증권
- 원산지증명서

[CATEGORY REFERENCE — do not include in output]
- 상업송장: Commercial invoice — export/import transaction amounts, item descriptions, seller/buyer info
- 포장명세서: Packing list — cargo packaging units, weights, quantities, dimensions
- 선하증권: Bill of Lading — maritime transport contract, cargo receipt, title document
- 원산지증명서: Certificate of Origin — proof of country of origin for export goods

[OUTPUT FORMAT — STRICT]
Output the category name as PLAIN TEXT on a single line. Nothing else.
The following are forbidden: markdown syntax (no #, *, **, _, `, >, -, +), code fences, HTML tags, emoji, quotation marks, prefixes such as "Category:" or "Answer:", explanations, reasoning, confidence labels, leading or trailing blank lines, leading or trailing whitespace, and punctuation at the end.
The output must be EXACTLY one of the following:
- One of the four existing category names, written verbatim as listed in [CATEGORY LIST].
- A newly proposed category name following the naming rules above.
- The literal string: Unclassifiable
```

#### 한국어 출력 프롬프트

Category-Only와 동일한 구조에서 **출력 언어만 한국어로 변경**한 버전이다. `[ROLE]`도 한국어 출력 명시로 변경되며, Step 1 출력은 `분류 불가`, 새 카테고리는 한국어 2-6 단어로 제한된다.

> 상세 내용: `docs/week3/document_classifier_prompt_korean.md`

### 2.3. 2주차 대비 변경점

| 항목 | 2주차 | 3주차 |
|---|---|---|
| 출력 형식 | JSON (`{"label": "..."}`) | 순수 문자열 (카테고리명만) |

## 3. Edge Case 테스트를 위한 데이터 수집

### 3.1. 목적

2주차 은행 데이터(40장, 4카테고리)는 문서 제목에 카테고리명이 직접 포함되어 있어 난이도가 낮았다 (Qwen3-8B 한국어 100%).  
3주차에서는 **기존 카테고리에 완벽히 매칭되지 않는 문서**를 준비하여, 모델의 새 카테고리 생성 능력과 모호한 입력에 대한 분류 안정성을 테스트한다.

### 3.2. 수집 데이터: 물류 도메인

| 카테고리 | 장수 | 설명 |
|---|---|---|
| 상업송장 (Commercial Invoice) | 15장 | 수출입 거래 금액·품목 명세 |
| 포장명세서 (Packing List) | 11장 | 화물 포장 단위·중량·수량 명세 |
| 선하증권 (Bill of Lading) | 6장 | 해상운송 계약 증빙·화물 수령증 |
| 원산지증명서 (Certificate of Origin) | 7장 | 수출국 원산지 증명 |
| 기타 | 13장 | 상기 4종에 해당하지 않는 문서 (보험증권, 검사증명서 등) |
| **합계** | **52장** | |

경로: `data/OCR_sample/물류/`  
OCR 추출: PaddleOCR-VL-1.5 (`results/OCR/paddleOCR-VL/test.py`, GPU 추론, 52장 모두 정상 변환)

### 3.3. Edge Case 테스트 설계

물류 데이터를 **물류 4카테고리 프롬프트**에 입력하여 다음을 측정한다.

| 테스트 관점 | 측정 방법 |
|---|---|
| 기존 카테고리 분류 정확도 | 4 LABELS에 해당하는 39장(상업송장 15·포장명세서 11·선하증권 6·원산지증명서 7) 기준 Accuracy / per-class P·R·F1 |
| 새 카테고리 생성 품질 | LABELS 매칭 실패 시 모델이 제안한 카테고리명의 도메인 적합성·일관성·재사용성 |
| `기타` 문서 대응 | LABELS에 없는 13장이 (a) `분류 불가`, (b) 새 카테고리, (c) 잘못된 4-LABEL 중 어디로 분기하는지 분포 측정 |
| 모델 크기 영향 | Qwen3-8B vs Qwen3-4B를 동일 조건으로 비교하여 instruction following 차이 확인 |

> `기타` 13장은 모델이 직접 매칭할 수 있는 LABEL이 없으므로 정확도 개념이 성립하지 않는다. 따라서 4 LABELS에 한해 정량 평가하고, `기타`는 분포 분석으로 별도 측정한다.

## 4. 테스트 결과

### 4.1. 실험 환경

| 항목 | 값 |
|---|---|
| OCR 파이프라인 | PaddleOCR-VL-1.5 (`paddleOCR-VL/test.py`, GPU 추론, 52/52 성공) |
| 분류 모델 | Qwen3-8B / Qwen3-4B (Ollama Q4_K_M, `temperature=0.0`, `num_predict=256`, `think=False` + `/no_think` 디렉티브) |
| 테스트 셋 | 물류 도메인 52장 (상업송장 15·포장명세서 11·선하증권 6·원산지증명서 7·기타 13) |
| 프롬프트 | 영문(`document_classifier_prompt_category_only.md`) / 한국어(`document_classifier_prompt_korean.md`) 2종 |
| LABELS | 4종 (`상업송장`, `포장명세서`, `선하증권`, `원산지증명서`) |
| 평가 지표 | true ∈ 4 LABELS인 39장에 대한 Accuracy·per-class P/R/F1, true=`기타` 13장에 대한 분포 분석, 새 카테고리 제안의 일관성 |

### 4.2. 프롬프트 설계 반복 — 5-class → 4-class

처음에는 폴더 라벨 5종(`상업송장`, `포장명세서`, `선하증권`, `원산지증명서`, `기타`)을 그대로 LIST에 넣어 실험했다. 그 결과 8B는 한·영 모두 0.731(38/52)을 기록했으며, 4 주력 카테고리는 거의 만점이지만 **13장의 `기타` 문서를 단 한 번도 `기타`로 분류하지 않았다**. 한국어 프롬프트에서는 13장 전부 `분류 불가`, 영문 프롬프트에서는 12장 `Unclassifiable` + 1장 새 카테고리(`Marine Cargo Insurance Policy`)로 분기됐다.

원인은 `[CLASSIFICATION RULES]` 4-e 조항(`Avoid catch-all names such as "Other", "Miscellaneous", "General"`)과 LABELS의 `기타`가 충돌하기 때문이다. 모델은 `기타`를 명시적으로 금지된 catch-all로 해석해 회피하고 Step 1 또는 Step 3로 분기한다. 즉 8B의 27% 오답 전부가 모델 한계가 아닌 프롬프트의 내적 모순에서 발생했다.

이 진단을 반영해 `기타`를 LABELS에서 제거하고, 모델이 `기타`성 문서를 (a) `분류 불가`로 빠지거나, (b) 새 카테고리를 제안하도록 의도적으로 유도하는 4-class 구조로 프롬프트를 수정했다. `evaluate.py`도 4 LABELS 코어 평가와 `기타` 분포 평가를 분리하도록 재작성했다.

### 4.3. 4-class 전체 정확도 / 레이턴시

39장(true ∈ 4 LABELS) 기준 정량 결과.

| 모델 / 프롬프트 | Core Accuracy | 정답 수 | parse_fail | Avg Latency |
|---|---:|---:|---:|---:|
| **Qwen3-8B / 한국어 프롬프트** | **0.974** | 38 / 39 | 0 | 760 ms |
| **Qwen3-8B / 영문 프롬프트** | **0.974** | 38 / 39 | 0 | 770 ms |
| Qwen3-4B / 한국어 프롬프트 | 0.462 | 18 / 39 | 0 | 3,396 ms |
| Qwen3-4B / 영문 프롬프트 | 0.359 | 14 / 39 | 1 | 3,398 ms |

8B는 한·영 모두 0.974로 거의 만점이며, 응답 시간도 1초 미만으로 동일하다. 5-class 구조 대비 8B의 정확도가 0.731 → 0.974로 크게 상승했다.

| 상업송장 | 0.966 | 0.966 | 0.385 | 0.333 | 15 |
| 포장명세서 | 1.000 | 0.957 | 0.526 | 0.444 | 11 |
| 선하증권 | 1.000 | 1.000 | 0.500 | 0.154 | 6 |

8B는 4 카테고리 모두에서 0.957 이상의 F1을 기록했고, 유일한 오답은 두 프롬프트 모두에서 흔들린 `IMG_OCR_6_T_NV_003382` 1건이었다. OCR 결과 자체가 모호한 샘플로 추정된다.

### 4.5. `기타` 13장의 분포 — 한·영 프롬프트 차이

| 분기 | 8B / 한국어 | 8B / 영문 | 4B / 한국어 | 4B / 영문 |
|---|---:|---:|---:|---:|
| `Unclassifiable` / `분류 불가` | 12 (92.3%) | 4 (30.8%) | 0 | 3 (23.1%) |
| 새 카테고리 제안 | 1 (7.7%) | 8 (61.5%) | 0 | 8 (61.5%) |
| 4-LABEL 오분류 | 0 | 1 (7.7%) | 13 (100%) | 1 (7.7%) |
| Parse fail | 0 | 0 | 0 | 1 (7.7%) |

**8B는 한·영 프롬프트 사이에서 명확히 다른 전략을 선택한다**:

- **한국어 프롬프트**: 92.3%(12/13)를 `분류 불가`로 처리. 매우 보수적.
- **영문 프롬프트**: 61.5%(8/13)를 새 카테고리로 제안. 적극적.

한·영 모두 4-LABEL 오분류는 0 또는 1건뿐이라 **8B는 `기타`성 문서를 4-LABEL로 잘못 끌어들이지 않는다**(상업송장·포장명세서 등 명확한 문서의 분류 안정성을 해치지 않음).

반면 **4B/한국어는 13장 전부를 4-LABEL로 잘못 분류**하며 `분류 불가`나 새 카테고리 제안을 한 번도 사용하지 않는다 — Decision Tree의 Step 1·3을 무시하고 Step 2로 강제 매핑하는 패턴이다.

### 4.6. 새 카테고리 제안의 일관성 (Qwen3-8B / 영문)

8B/영문이 제안한 새 카테고리 8건은 4종으로 수렴한다.

| 제안된 카테고리명 | 빈도 | 도메인 적합성 |
|---|---:|---|
| Certificate of Origin | 4 | ✅ (단, 기존 LABEL `원산지증명서`와 의미 중복 — 영문 별칭 생성) |
| Marine Cargo Insurance Policy | 2 | ✅ |
| Certificate of Conformity | 1 | ✅ |
| Inspection Certificate | 1 | ✅ |

명명 규칙(2-6 단어, 명사구, 일반성)을 모두 만족하며 동일 유형 문서에 같은 이름을 재사용한다. 다만 `Certificate of Origin`은 기존 LABEL `원산지증명서`의 영문 번역에 가까워, 영문 출력에서는 모델이 한국어 LABEL을 영문 별칭으로 자연스럽게 재명명하는 부수 효과가 발생함을 확인할 수 있다.

8B/한국어는 `검사증명서` 1건만 새로 제안했고, 나머지 12건은 모두 `분류 불가`로 처리해 보수적 운영 정책에 적합하다.

|---|---:|---:|---:|---:|---:|---:|
| 상업송장 | 14 | 0 | 0 | 0 | 1 | 0 |
| 포장명세서 | 0 | 11 | 0 | 0 | 0 | 0 |
| 선하증권 | 0 | 0 | 6 | 0 | 0 | 0 |
| 원산지증명서 | 0 | 0 | 0 | 7 | 0 | 0 |

## 5. Instruction Following 테스트 (4B vs 8B)

### 5.1. 4-class 분류 성능 비교

| 지표 | Qwen3-8B (한국어) | Qwen3-4B (한국어) | 차이 |
|---|---:|---:|---:|
| Core Accuracy | 0.974 | 0.462 | **−0.512** |
| Avg Latency | 760 ms | 3,396 ms | **+4.5×** |
| Parse fail | 0 | 0 | 0 |

| 지표 | Qwen3-8B (영문) | Qwen3-4B (영문) | 차이 |
|---|---:|---:|---:|
| Core Accuracy | 0.974 | 0.359 | **−0.615** |
| Avg Latency | 770 ms | 3,398 ms | **+4.4×** |
| Parse fail | 0 | 1 | +1 |

### 5.2. 4B의 Instruction Following 실패 패턴

**1) 4B/한국어 — Step 1·3을 무시하고 Step 2로 강제 분기**

4B/한국어는 `기타` 13장을 전부 4-LABEL로 잘못 매핑했다. 분포는 `원산지증명서` 6건(46.2%), `선하증권` 5건(38.5%), `상업송장`·`포장명세서` 각 1건. Decision Tree의 Step 1(Ambiguity Check)·Step 3(New Category Proposal) 분기를 한 번도 발동하지 않았다.

이 패턴은 4 LABELS 분류에도 영향을 미쳐, 명확한 문서를 다른 LABEL로 끌어들이는 오분류가 빈번하다 (예: `포장명세서`→`원산지증명서` 6건, `상업송장`→`원산지증명서` 5건).

**2) 4B/영문 — 새 카테고리 제안이 reasoning 단편으로 오염**

4B/영문은 새 카테고리를 11종 제안했지만, 대부분 명명 규칙을 위반한 reasoning 단편이다.

| 4B/영문 새 카테고리 (11종) | 분류 |
|---|---|
| `상`, `Total: "THREE HUNDRED NINETY FOUR(394) P`, `THE NUMBER OF CREDIT 2148/382862` 등 | OCR 본문 단편 직접 복사 |
| `The document has`, `The document has clear`, `The document is clearly about an inspection`, `Action verbs: ...` | reasoning 미완 단편 |
| `For example: "83-14" (Item Number), "FILTER-FUEL PUMP" (Description` | OCR 예시 텍스트 그대로 |

대조적으로 8B/영문이 제안한 4종(`Certificate of Origin`, `Marine Cargo Insurance Policy`, `Certificate of Conformity`, `Inspection Certificate`)은 모두 명명 규칙을 만족하고 도메인 적합하다.

**3) 응답 시간 4.5배 — `/no_think` 부분 적용**

`/no_think` 디렉티브와 `think=False` 옵션을 동시에 적용했음에도 4B는 8B 대비 평균 4.5배 느렸다. 1차 실험에서는 reasoning preamble("Okay, let's tackle this classification problem...")이 출력에 그대로 노출되어 `num_predict=32`로 잘려 정답에 도달하지 못했고, `num_predict=256`으로 늘리고 파서를 `<think>` 블록 제거 + 마지막 줄 매칭으로 강화한 후에야 답변을 회수할 수 있었다. 즉 4B는 thinking 모드 제어 명령을 부분적으로만 따른다.

### 5.4. 결론 — 4B vs 8B

| 평가 항목 | 결과 |
|---|---|
| **정확도** | 8B 압승 (0.974 vs 0.462, 영문 격차는 0.615) |
| **응답 속도** | 8B가 4.5배 빠름 (4B의 reasoning 잔존이 주 원인) |
| **Instruction Following** | 8B는 한국어 LABEL 출력·새 카테고리 생성 규칙·Decision Tree 분기를 모두 정확히 따름. 4B는 한국어에서 Step 1·3 무시, 영문에서 reasoning 단편을 새 카테고리로 출력하는 등 지시 위반 빈번 |
| **가성비** | VRAM 절감(약 2.7GB)을 고려해도, 정확도·속도 양면에서 격차가 커 4B를 운영 모델로 채택하기 어려움 |

→ **MVP 운영 모델로는 Qwen3-8B를 채택**하고, 4B는 후속 단계의 양자화·파인튜닝 검증용으로만 검토한다.

→ **운영 시 한·영 프롬프트 정책**: 보수적 운영(미분류 문서를 사람이 검토)이 목적이면 한국어 프롬프트(`분류 불가` 92.3%), 자동 카테고리 확장이 목적이면 영문 프롬프트(새 카테고리 제안 61.5%)를 선택할 수 있다.

## 6. 결론 및 후속 과제

### 6.1. 3주차 성과

- 멘토 피드백을 반영하여 프롬프트를 "카테고리명만 출력"하는 구조로 재설계 (3종)
- Decision Tree(Ambiguity → Match → New) 강제, 새 카테고리 생성 규칙 구체화
- 물류 도메인 52장 데이터 수집·OCR 변환(PaddleOCR-VL-1.5)으로 Edge Case 테스트 기반 구축
- 5-class → 4-class 프롬프트 반복으로 catch-all 충돌 문제 진단·해결, **Qwen3-8B 한·영 프롬프트 모두에서 Core Accuracy 0.974 달성**
- `evaluate.py`를 4 LABELS 코어 평가 + `기타` 분포 + 새 카테고리 일관성으로 분리해 Edge Case 정량 평가 인프라 구축

### 6.2. 후속 과제

- **VRAM 벤치마크 측정**: `nvidia-smi`/`ollama ps`로 8B·4B의 실제 점유율 기록 (5.4 TODO)
- **OCR 모호 샘플 분석**: 두 프롬프트 모두에서 흔들린 `IMG_OCR_6_T_NV_003382` 1건의 OCR 결과를 검토하여 OCR 모델 변경(GLM-OCR, DeepSeek-OCR-2)이 분류 정확도에 미치는 영향 측정
- **새 카테고리 일관성 강화**: 8B/영문이 `Certificate of Origin`을 4회 재사용한 사례처럼, 새 카테고리 명명을 운영 단계에서 정규화하는 후처리 로직 검토
- **MVP FastAPI 서비스 통합**: Qwen3-8B + 한국어 프롬프트(보수적) / 영문 프롬프트(적극적)를 운영 정책 옵션으로 노출하는 API 설계
